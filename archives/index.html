07. 제이쿼리 모바일 버튼과 리스트뷰

  • 제이쿼리 모바일 버튼

  • 버튼 생성 3가지 방법

    • a 태그 - 가장 많이 사용됨
    • button태그
    • input태그

  • 3가지 버튼 생성 방법

    • 태그와 명세 방법은 다르지만 생선된 버튼의 모양은 모두 같음
    • 버튼들은 둥근 모서리와 테두리 음영 효과가 적용
    • 버튼 크기는 화면 전체의 너비나 문자열 길이에 비례
    • 버튼 문자열은 가운데 정렬

  • 링크 버튼

    • 버튼처럼 보여지는 링크로 가장 많이 사용
    • 태그는 링크 문자열 형태로 표시는 터치하기 불편해서 모바일 장치에 부적합

  • 헤더나 푸터 영역

    • 링크 태그만으로 버튼 생성 가능

  • 콘텐츠 영역

  • 폼 버튼

    • 입력이나 수정 등 폼과의 상호 작용이나 완성된 폼의 제출을 위해 사용
    • HTML5 표준 태그인 button 혹은 input 태그를 그래도 사용
    • 추가할 속성도 없고 간단
    • 모바일 버튼으로 자동 변환
    • 고전적인 버튼 모양으로 표시

  • 아이콘 버튼

    • 40개의 표준 아이콘을 기본적으로 제공

    • 문자열 생략

      • 페이지에 버튼이 많을 경우 버튼 크기를 줄이기 위해 문자열 없이 아이콘만 버튼에 표시
      • a, button, input 태그 모두에 설정이 가능

    • 아이콘 위치 변경

      • 기본적으로 아이콘은 버튼의 왼쪽에 정렬되어 표시
      • 버튼 내에서 아이콘의 위치 변경 가능

    • 사용자 정의 아이콘

      • 버튼 생성 마크업 안에 data-icon 속성값으로 고유한 아이콘 이름(식별자)를 임의로 생성하여 지정

      • 추가하는 아이콘의 이름은 미리 정의되어야 함

      • style 태그 안에 사용자정의 아이콘의 파일위치와 아이콘 스타일 속성들로 CSS3클래스를 정의

  • 이미지 버튼

    • 아이콘이나 문자열 대신에 이미지만으로 버튼 생성
    • 모바일 환경을 고려하여 .png파일을 경로명을 src 속성값으로 지정

  • 버튼 그룹

    • 버튼들을 하나의 컨트롤처럼 보이도록 묶어서 버튼 그룹 생성
    • 버튼 간의 여백 없이 버튼들을 모아서 표시함으로써 의미적으로 밀접하게 관련된 버튼들을 그룹화할때 사용

  • 버튼 테마

    • 스타일 기본 테마는 a~b까지 2가지 종류
    • 버튼도 다른 화면 요소들처럼 상위 컨테이너의 테마를 상속받음
    • 버튼마다 data-theme 속성값을 설정함으로써 각 버튼의 테마 변경 가능

  • 리스트뷰

    • 여러 리스트들 즉, 목록들을 모아 간결한 형태로 화면에 표시
    • 설정된 링크를 따라 페이지로 이동하거나 항목에 대한 상세 정보를 확인할 수 있음
    • 제이쿼리 모바일에서는 다양한 스타일의 리스트뷰를 쉽고 빠르게 생성할 수 있으므로 대표적인 브라우징 요소로 많이 사용
    • 제이쿼리 모바일 페이지들은 대부분 리스트뷰 기반의 레이아웃을 갖음
    • 제한된 화면 공간에 필요한 정보만을 요약, 제공할 때 매우 효과적
    • 정보 유형이나 활용 목적에 따라 다양한 종류의 리스트부를 선택 사용

  • 순서 리스트뷰

    • 자동으로 항목마다 번호가 앞에 표시되는 리스트뷰

  • 연결 리스트뷰

    • 항목 선택이 가능
    • 기본 리스트뷰의 각항목에 링크 정보를 추가(각 항목에 오른쪽 화살표 아이콘이 추가)
    • 링크를 통해 특정 페이지 이동을 안내, 다른 하위 리스트뷰를 연결하기 위해 사용

  • 인셋 리스트뷰

    • 콘텐츠 영역 안에 둥근 사각형 모양의 내부 블록을 형성하고 그 안에 리스트뷰를 표시
    • 둥근 리스트뷰

  • 분할 리스트뷰

    • 각 항목 영역을 분리하여 둘 이상의 링크를 통해 여러 페이지와의 연동이 가능한 리스트뷰
    • 분할된 항목 영영ㄱ은 독립된 분할 버튼으로 동작

  • 카운트 버블

    • 리스트뷰 항목의 둥근 거품 방울 모양의 화면 요소

  • 리스트 구분자

    • 원하는 항목을 찾기 쉽도록 각 그룹별로 분류명을 표시하는 특별한 항목을 추가
    • 항목 그룹에 대한 ‘헤더’와 같은 역할

  • 리스트뷰 제목 생성하기

    • 항목들이 하나의 그룹으로만 분류되는 경우, 리스트 구분자는 항목들의 제목 역할을 함
    • 메뉴 화면 작성시 많이 사용

  • 썸네일 리스트뷰

    • 썸네일 이미지를 항목 왼쪽에 추가한 리스트뷰
    • 썸네일 이미지: 데이터 전송의 효율성을 위해 변환된 작은 크기의 이미지

  • 아이콘 리스트뷰

    • 각 항목 왼쪽에 썸네일보다 작은 이미지를 아이콘처럼 표시

  • 여러 줄 리스트뷰

    • 표시 내용의 중요도에 따라 강조할 내용은 제목 태그로, 나머지 내용은

      문단 태그를 사용하여 일정한 형식으로 상세한 설명과 관련 정보들을 여러 줄에 표시

  • 검색 필터 리스트

    • 항목들의 맨위쪽에 검색 필터 상자를 추가
    • 원하는 항목만 직접 키워드 검색 방식으로 일치하는 항목만 걸러서 보여줌

  • 페이지 탭

    • 페이지 공간 안에 여러 개의 탭 페이지를 생성 각 탭 페이지를 선택저긍로 브라우징하는 인터페이스
    • 모바일의 화면의 좁은 페이지 공간을 효율적으로 활요하는 방법 중 하나
    • 보통 현재 페이지와 의미적으로 연관된 복잡하고 다양한 정보들을 제공할 때 사용한다.

  • 컬럼 토글 테이블

    • 좁은 공간을 위해 테이블 컬럼을 선택적으로 표기하거나 숨실 수 있는 테이블
    • 숨겨진 컬럼들은 표시를 원할 경우, 컬럼 팝업메뉴를 통해 선택함으로써 표시
    • 화면 폭을 고려하여 컬럼 우선순위를 부여함으로써 표시 테이블 컬럼 항목들을 제한
    • 모바일 화면의 크기 허용 범위 안에서 우선 순위대로 최대한 컬럼들이 표시되고 우선 순위가 낮은 컬럼들은 표시되지 않고 숨겨진다.

  • 숨김 패널

    • 페이지 영역을 확장할 수 있는 효과적인 방법
    • 숨심 패널 안에 매뉴나 폼 등을 숨겨 놓았다가 필요할 때 페이지를 전환하지 않고 제공

08. 제이쿼리 모바일 대화상자와 폼

  • 대화상자
    • 제이쿼리 모바일이 지원하는 팝업 창 형태의 특별한 페이지
    • 둥근 모서리와 바깥 여백, 그람자 등의 스타일이 적용되어 기존 페이지 위에 표시
    • 확인 창이나 경고 창, 액션시트 스타일들의 유형을 만들때 사용
  • 액션시트
    • 헤더와 푸터가 없는 특별한 대화상자
    • 보통 사용자에게 의견을 붙고 응답을 요청하는 경우에 사용된다.
    • 취소를 포함한 선택 가능한 작업 항목들을 제시할 목적으로 사용되므로 보통 둘 이상의 버튼을 포함한다.
    • 헤더와 푸터가 없기 때문에 대화상자를 닫을 수 있는 링크 버튼이 꼭 필요하다.
  • 페이지 테마
  • 콘텐츠 레이아웃
    • 스마트폰은 와면 크기가 작고 폭이 좁기 때문에 모바일 페이지의 레이아웃 구성이 중요
    • 바 레이아웃
      • 콘텐츠 영역 안에 제목처럼 한 줄 단위의 정보를 표기하기 위한 레이아웃
    • 블록 레이아웃
      • 콘텐츠 영역 안에 사각형 모양의 박스 영역을 생성하여 내용들을 규칙적으로 배치하거나 특정 내용으 강조하는 레이아웃
    • 그리드(격자) 레이아웃
      • 그리드 레이아웃은 둘 이상의 블록들을 갖는 그리드 컨테이너를 만듦으로써 생성됨
      • 그리드 컨테이너
      • 그리드 블록
  • 접이식 패널 레이아웃
    • 콘텐츠의 특정 영역을 접혔다 펼치기를 반복하며 내용을 숨기거나 표시 할 수 있다.
    • 스마트폰의 좁은 화면 영역을 효과적으로 사용할 수 있는 방법
    • 콘텐츠 영역을 접이식 패널을 여러개 배치 가능
    • 기본 접이식 패널
    • 중첩 접이식 패널
      • 접이식 패널 안에 또 다른 접이식 패널을 포함
      • 중첩 단계 2~3단계가 적당(제한은 없다)
    • 그룹 접이식 패널
      • 여러 접이식 패널을 하나의 그룹으로 묶은 것으로 하나의 연결된 요소로 동작함
      • 새로운 접이식 패널을 펼치게 되면 이전에 펼쳐졌던 패늘은 자동으로 접힘

    • 마크업 문서의 스타일을 정의하는 속서들을 모아 놓은 언어이기 때문에 HTML문서뿐만 아니라 XML문서에도 적용가능
    • 화면 구성이나 표현 형식 등 디자인 부분은 HTML대신에 CSS에 위임함으로써 문서 구조와 문서표현을 자연스럽게 분리할 수 있다.

  • CSS 사용의 이점

    • 확장성
    • 편의성
    • 재사용성 : 독립된 스타일 모듈을 작성, 여러 HTML 문서에 공통으로 활용
    • 생산성 : 모듈 단위의 협업과 생산성 향상이 가능

  • CSS 스타일 선언 형식

    • 선택자
      • 스타일을 적용할 대상을 지정
      • 특정 태그 영역만 원하는 스타일 또는 특별한 기능(제이쿼리 사용)을 적용
    • 스타일 속성 블록
      • 선택자에 의해 선택된 영역에 적용할 색상, 크기 등의 구체적인 스타일을 명세
      • 스타일 속성 블록은 중괄호로 둘러싸며 하나 이상의 스타일 속성 선언을 포함
    • 스타일 속성 이름
      • 선택자에 대해 어떤 스타일을 부여할지 적용할 실질적인 기능에 해당하는 이름을 지정(기능을 쉽게 유추할 수 있는 단어로 구성)
    • 스타일 속성 값
      • 스타일 속성의 종류에 따라 설정 값의 유형과 범위가 제한됨
      • 의미를 표현하는 키워드나 숫자를 사용

  • 클래스 선택자는 “.”을 사용하고 아이디 선택자는”#”을 사용

  • 스타일시트 선언 방식

    • 내부 방식 : 실무에서 거의 사용하지 않음, HTML5문서 내부에 스타일시트를 삽입하는 형식(임베디드 형식)

    • 외부 방식 : 실무에서 가장 많이 사용, HTML5 문서 외부에 독립된 스타일시트 파일을 작성하여 연결하는 형식(링크 형식)

    • 인라인 방식 : HTML5 문서 안의 각 엘리먼트에 속성으로 스타일을 끼어 넣는 형식

    • 임포트 방식 : 스타일시트 안에서 또 다른 스타일시트 파일을 포함시키는 방식

      => 실무에서는 외부 방식과 인라인 방식을 혼합하여 제일 많이 사용함

  • 스타일 적용 우선수위(낮음 => 높음)

    • 임포트 => 외부 방식 => 내부 방식 => 인라인 방식
    • 태그 선택자 => 계층 선택자 => 클래스 선택자 => 아이디 선택자

###소프트웨어 공학

소프트웨어 개발의 생명주기(waterfall process)

  • 계획, 개발, 시험, 운용하는 전체과정
  • Requirement Analysis => System speccification => Design => Implementation => verification =>
  • 각 단계가 완료되어야 다음 단계로 넘어갈 수 있음, 모든 구성원이 각 단계에 모두 참여함, 완벽을 중요시하고 인원이 많은 프로젝트에 적합

Agile Development Process(에자일)

  • Code-oriented Process / 한마디로 정의한다면!!!
  • 각 단위가 독립적으로 작용함
  • 일정한 주기를 반복 수행하면서 prototype를 발전 시켜 나감
  • 각자가 유기적으로 움직여야 잘 운용됨
  • 스크럼(일 단위 각자 할일을 공유함) ex) 트렐로 / 코드리뷰와 페어프로그래밍

에자일 프로세스의 진행 순서
